“How do I use the latest version of Python package alpaca-trade-api?”¶

Check the latest version of Python package alpaca-trade-api here:

  • PyPI

  • GitHub Releases

To check the student’s version of Python package alpaca-trade-api for a given Conda environment, run the following terminal command in that Conda environment:

conda list alpaca

# packages in environment at /opt/anaconda3:
#
# Name                    Version                   Build  Channel
alpaca-trade-api          2.1.0                    pypi_0    pypi

Here’s an example of the code required to make an API call in the latest version of Python package alpaca-trade-api–version 2.1.0 as of Apr 27, 2022–which can be run in a single Jupyter cell:

import pandas as pd
from alpaca_trade_api.rest import REST, TimeFrame
from dotenv import find_dotenv, get_key

alpaca_api_key = get_key(find_dotenv(), 'ALPACA_API_KEY')
alpaca_sec_key = get_key(find_dotenv(), 'ALPACA_SEC_KEY')

api = REST(
  alpaca_api_key,
  alpaca_sec_key,
  api_version='v2',
)

tickers    = ['AGG', 'SPY']
timeframe  = TimeFrame.Day  # '1Day'
start_date = '2020-08-07'   #pd.Timestamp('2020-08-07', tz='America/New_York').isoformat()
end_date   = '2020-08-07'   #pd.Timestamp('2020-08-07', tz='America/New_York').isoformat()

portfolio_prices_df = api.get_bars(
  tickers,
  timeframe,
  start=start_date,
  end=end_date,
).df
portfolio_prices_df

“I receive the following error message when I attempt to connect to PostgreSQL via SQLAlchemy: OperationalError: (psycopg2.OperationalError) could not translate host name "...@127.0.0.1" to address: unknown server error”¶

This problem arises when the student-set password for the default PostgreSQL user postgres includes a special character such as the at sign @.

To solve this problem, either the student may change the password for the default PostgreSQL user postgres to one which does not include special characters such as the at sign @ or the student may percent-encode their current password as follows:

config.py

db_password = 'myPassword@'

movies.ipynb

from sqlalchemy import create_engine
from config import db_password
import urllib.parse

db_string = f'postgresql://postgres:{urllib.parse.quote(db_password)}@127.0.0.1:5432/movie_data'
engine    = create_engine(db_string)
movies_df.to_sql(name='movies', con=engine)

What follows is context surrounding this problem:

The document on SQLAlchemy Database URLs states,

“As the URL is like any other URL, special characters such as those that may be used in the password need to be URL encoded to be parsed correctly.. Below is an example of a URL that includes the password “kx%jj5/g”, where the percent sign and slash characters are represented as %25 and %2F, respectively:

postgresql+pg8000://dbuser:kx%25jj5%2Fg@pghost10/appdb

The encoding for the above password can be generated using urllib.parse:

>>> import urllib.parse
>>> urllib.parse.quote_plus("kx%jj5/g")
'kx%25jj5%2Fg'

The document on PostgreSQL Connection Strings states,

“The connection URI needs to be encoded with percent-encoding if it includes symbols with special meaning in any of its parts. Here is an example where the equal sign (=) is replaced with %3D and the space character with %20:”.

“import column error”¶

  1. Make sure the order of the SQL columns matches the columns of the csv file for that table

  2. If a student is having a problem running a query make sure there are no double quotes inside the query. Switch double quotes inside the query to single quotes. The Reason Why

  3. Try an insert query rather than csv import since the errors are more meaningful.

“I receive the error message [Errno 1]: Operation not permitted: '/Users/username/Desktop' when I attempt to access my Desktop from within pgAdmin and there is a red lock icon next to my Desktop folder icon in pgAdmin.”¶

This problem appears to have a relation to the user permissions including which parts of the filesystem the user is allowed to read from and write to.

On macOS Monterey, navigate to the following setting and verify the pgAdmin has the correct permissions:

  • System Preferences > Security & Privacy > Privacy > Accessibility

  • System Preferences > Security & Privacy > Privacy > Files and Folders

“I receive a ModuleNotFoundError when I run import x even though I already installed Python package x.”¶

This scenario is often the result of the inconsistent use of conda environments across tools such as Git Bash, Jupyter, and VS Code.

Within each relevant tool, run

conda env list && conda list x

Is the output consistent? If not, then change the conda environment or Python interpreter of whichever tool produces the error message to that environment in which the student claims they have installed Python package x.

“I receive a NameError even though I already defined my variable name.”¶

Names that have been defined belong to the namespace.

To check the namespace in Jupyter, for example, in a new cell, run

%who

If the name isn’t in the namespace, then it isn’t defined.

It often helps when the student restarts their notebook’s kernel and re-runs the cells from the top of the notebook, in order.

“I receive a FileNotFoundError when I try to read the file.”¶

Reasons for this error message may include:

  • A poor understanding of how to navigate the filesystem and of how to properly specify a relative path. Take this time to review these concepts with the student.

  • An operating system-specific syntax requirement. Use Python package pathlib to address this.

“I receive the following error message when I make an API call to the Alpaca API with my API key: UnicodeEncodeError: 'latin-1' codec can't encode character '\u201c' in position 0: ordinal not in range(256).”¶

TL;DR¶

Open the student’s file .env and remove the pair of slanted quotation marks around their API keys; save the file .env; and re-run the program .py/.ipynb from the top. This should resolve the UnicodeEncodeError.

Description¶

In the Alpaca API unit in which students place their API key in a config file called .env, it is common to find the student’s API key wrapped in a pair of slanted quotation marks:

ALPACA_API_KEY=“abcdefg”
ALPACA_SEC_KEY=“hijklmnopqrstuvwxyz”

The student’s machine’s settings may be such that the default quotation marks placed in text files is a pair of slanted quotation marks “ and ” which in Unicode are identified by the code points U+201C and U+201D and the names LEFT DOUBLE QUOTATION MARK and RIGHT DOUBLE QUOTATION MARK, respectively, instead of the vertical quotation mark " which in Unicode is identified by the code point U+0022 and the name QUOTATION MARK:

import unicodedata

points = [
  '\u201C',
  '\u201D',
  '\u2018',
  '\u2019',
  '\u0022',
  '\u0027',
  '\u0060',
  '\u00B4',
]

print(f"{'Symbol':<10} {'Code Point':<10} {'Category':<10} {'Name':<10}")
for point in points:
  print(f"{chr(ord(point)):<10} "
        f"{format(ord(point), '04x'):<10} "
        f"{unicodedata.category(point):<10} "
        f"{unicodedata.name(point)}")

Symbol     Code Point Category   Name
“          201c       Pi         LEFT DOUBLE QUOTATION MARK
”          201d       Pf         RIGHT DOUBLE QUOTATION MARK
‘          2018       Pi         LEFT SINGLE QUOTATION MARK
’          2019       Pf         RIGHT SINGLE QUOTATION MARK
"          0022       Po         QUOTATION MARK
'          0027       Po         APOSTROPHE
`          0060       Sk         GRAVE ACCENT
´          00b4       Sk         ACUTE ACCENT

The Python package python-dotenv documentation states

“Keys can be unquoted or single-quoted. Values can be unquoted, single- or double-quoted. Spaces before and after keys, equal signs, and values are ignored. Values can be followed by a comment. Lines can start with the export directive, which has no effect on their interpretation.”

This means that quotation marks aren’t necessary. Here’s an example of what the contents of the file .env should look like:

ALPACA_API_KEY=abcdefg
ALPACA_SEC_KEY=hijklmnopqrstuvwxyz

“My PyViz environment was working fine before, but now it isn’t. I didn’t make any changes, so why would it just stop working?”¶

This problem may arise when some subset of packages in the environment are upgraded (or downgraded) to different versions. The key to resolving this problem seems to be to find some set of package versions which all work well together.

Recreate the PyViz environment with the required package versions and then pin the packages the versions of which one wishes to remain constant so that Conda doesn’t update some subset of those packages the next time a new package is installed.

As a preemptive measure, the student may want to consider creating one or more duplicates of a functioning environment as a backup when they happen upon a mutually-compatible set of package versions. Duplicate environments may also serve as labs where the student can experiment with different package versions and scrap broken environments without worrying about having to start over from scratch.

Here’s an example procedure:

conda create -n pyvizenv python=3.7 anaconda nb_conda -y
conda install -c conda-forge jupyterlab=2.2 nodejs=12 -y
conda install -c pyviz holoviz hvplot geoviews -y
conda install -c plotly plotly -y
pip install numpy==1.19
pip install matplotlib==3.0.3

export NODE_OPTIONS=--max-old-space-size=4096 (macOS)
set NODE_OPTIONS=--max-old-space-size=4096 (Windows)

jupyter labextension install @jupyter-widgets/jupyterlab-manager --no-build
jupyter labextension install jupyterlab-plotly --no-build
jupyter labextension install plotlywidget --no-build
jupyter labextension install @pyviz/jupyterlab_pyviz --no-build

jupyter lab build

unset NODE_OPTIONS (macOS)
set NODE_OPTIONS= (Windows)

echo "jupyterlab ==2.2" >> ~/opt/anaconda3/envs/pyvizenv/meta-conda/pinned
echo "nodejs ==12" >> ~/opt/anaconda3/envs/pyvizenv/meta-conda/pinned
echo "numpy ==1.19" >> ~/opt/anaconda3/envs/pyvizenv/meta-conda/pinned
echo "matplotlib ==3.0.3" >> ~/opt/anaconda3/envs/pyvizenv/meta-conda/pinned
